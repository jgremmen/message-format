# Plan: Vollständige Pygments-Lexer-Implementierung für MessageFormat-Grammatik

Ein vollständiger Pygments-Lexer wird von Grund auf implementiert, der die MessageFormat-Grammatik aus `MessageLexer.g4` und `MessageParser.g4` exakt abbildet. Alle Token-Zuordnungen folgen strikt `tokens.md`, wobei Parser-Regeln als Token-Sequenzen verstanden werden.

## Steps

1. **Implementiere State-Architektur mit 11 States**: Erstelle [message_format.py](syntax/pygments/message_format.py) als `RegexLexer` mit States: `root` (DEFAULT-Modus), `parameter-name`, `parameter-body`, `parameter-format-colon`, `parameter-format-name` (PARAMETER-Modus mit Position-Tracking), `template-name`, `template-body` (TEMPLATE-Modus), `postformat-name`, `postformat-message`, `postformat-config` (POST_FORMAT-Modus), `single-quote`, `double-quote` (TEXT1/TEXT2-Modi).

2. **Implementiere Root-State mit Text-Tokenisierung**: Im `root` State erkenne `%{` → `Punctuation.Special` (P_START) + push `parameter-name`, `%[` → `Punctuation.Special` (TPL_START) + push `template-name`, `%(` → `Punctuation.Special` (PF_START) + push `postformat-name`. Erkenne `EscapeSequence` (\\u0000-\\uFFFF oder \\["%'{\\[]) → `String.Escape`, `CtrlChar` (\\u0000-\\u001F) → `Whitespace` (skip gemäß MessageLexer.g4 Zeile 49). Alle anderen `Character` (Unicode Zs/L/N/P/S) → `String` für `text` gemäß [tokens.md](tokens.md) Zeile 4.

3. **Implementiere Parameter-States mit mapKeys/configDefinition-Unterscheidung**: In `parameter-name` State erkenne erstes NAME/BOOL/NULL/EMPTY/FORMAT → `Name.Variable` (`parameterName` per [tokens.md](tokens.md) Zeile 19), dann `}` → `Punctuation.Special` (P_END) + pop, oder COMMA → `Punctuation` + pop zu `parameter-body`. In `parameter-body` erkenne: (1) **parameterFormat-Sequenz**: Keyword `format` → `Keyword.Reserved` (FORMAT per [tokens.md](tokens.md) Zeile 21) + pop zu `parameter-format-colon`, dort `:` → `Punctuation` (COLON) + pop zu `parameter-format-name`, dort NAME/BOOL/NULL/EMPTY → `Name.Function` (nameOrKeyword per [tokens.md](tokens.md) Zeile 22) + pop zurück zu `parameter-body`. (2) **mapKeys**: Operatoren (=, <>, !, <, <=, >, >=) → `Operator`, BOOL/NULL/EMPTY → `Keyword.Constant`, NUMBER → `Number.Integer`, `quotedString` via quotes, L_PAREN/R_PAREN → `Punctuation`. (3) **configDefinition**: NAME (kebab-case, keine Keywords) → `Name.Attribute`, COLON → `Punctuation`, dann BOOL → `Keyword.Constant`, NUMBER → `Number.Integer`, `simpleString`/`quotedMessage` via quotes oder keywords. Unterscheidung: NAME vor COLON ist immer `configDefinition` (da BOOL/NULL/EMPTY nicht als NAME erscheinen können), gemäß [MessageParser.g4](MessageParser.g4) Zeilen 67-133.

4. **Implementiere Template-States mit templateParameterDelegate-Sequenz**: In `template-name` State erkenne erstes NAME/BOOL/NULL/EMPTY → `Name.Label` (`templateName` per [tokens.md](tokens.md) Zeile 23), dann `]` → `Punctuation.Special` (TPL_END) + pop, oder COMMA → `Punctuation` + pop zu `template-body`. In `template-body` erkenne: (1) **templateParameterDelegate-Sequenz**: `simpleString` (nameOrKeyword oder `quotedString`) → bei nameOrKeyword tokenisiere NAME/BOOL/NULL/EMPTY als `Name.Variable` (erstes Token der Sequenz per [tokens.md](tokens.md) Zeile 24), bei `quotedString` tokenisiere Quotes als `String.Single`/`String.Double` und Inhalt als `String`, dann `=` → `Operator`, dann zweites `simpleString` analog als `Name.Variable` oder quoted. (2) **configDefinition**: NAME → `Name.Attribute`, COLON → `Punctuation`, dann Werte wie in Parameter, gemäß [MessageParser.g4](MessageParser.g4) Zeilen 81-112.

5. **Implementiere Post-Format-States**: In `postformat-name` State erkenne erstes NAME/BOOL/NULL/EMPTY/FORMAT → `Name.Function` (`postFormatName` per [tokens.md](tokens.md) Zeile 25), dann COMMA → `Punctuation` + pop zu `postformat-message`. In `postformat-message` State warte nur auf `'` → `String.Single` + push `single-quote` oder `"` → `String.Double` + push `double-quote` für zwingend erforderliches `quotedMessage`, nach Quote-Ende optional COMMA + pop zu `postformat-config` oder `)` → `Punctuation.Special` (PF_END) + pop komplett. In `postformat-config` State: NAME → `Name.Attribute`, COLON → `Punctuation`, BOOL/NULL/EMPTY → `Keyword.Constant`, NUMBER → `Number.Integer`, quotes für Strings/Messages, gemäß [MessageParser.g4](MessageParser.g4) Zeilen 98-112, [MessageLexer.g4](MessageLexer.g4) Zeilen 210-229.

6. **Implementiere Quote-States mit vollständiger Verschachtelung**: In `single-quote` State erkenne `'` (SQ_END) → `String.Single` + pop, `%{` → `Punctuation.Special` + push `parameter-name`, `%[` → `Punctuation.Special` + push `template-name`, `%(` → `Punctuation.Special` + push `postformat-name`, `EscapeSequence` → `String.Escape`, `CtrlChar` → `Whitespace` (skip), alle anderen `Character` → `String` (`text` per [tokens.md](tokens.md) Zeile 4). In `double-quote` State analog mit `"` (DQ_END) → `String.Double`. Start-Quotes werden beim Eintritt in die States als `String.Single`/`String.Double` tokenisiert gemäß [tokens.md](tokens.md) Zeilen 5-10, basierend auf [MessageLexer.g4](MessageLexer.g4) Zeilen 56-95.

7. **Erweitere Test-Suite für vollständige Coverage**: Füge zu [test_messageformat_lexer.py](test_messageformat_lexer.py) Tests hinzu: (1) MSG-010 `%(clip,'%{s,empty:x}',clip:34)` validiert `%(` → `Punctuation.Special`, `clip` → `Name.Function`, verschachteltes `%{s,empty:x}` in quotes mit `empty` → `Keyword.Constant`, zweites `clip` (config) → `Name.Attribute`, `34` → `Number.Integer`. (2) MSG-007 `format:mychar` validiert **3-Token-Sequenz**: `format` → `Keyword.Reserved`, `:` → `Punctuation`, `mychar` → `Name.Function`. (3) `templateParameterDelegate` `param=delegated` validiert **3-Token-Sequenz**: erstes `simpleString` (wenn NAME) → `Name.Variable`, `=` → `Operator`, zweites `simpleString` (wenn NAME) → `Name.Variable`. (4) MSG-004/006 `mapKeys` mit Operatoren (`!empty`, `<10`, etc.) und Gruppen `(key1,key2)`. (5) Alle vier `configDefinition`-Typen mit NAME → `Name.Attribute`. (6) Verschachtelte Post-Formats. (7) Edge-cases für `nameOrKeyword` (BOOL/NULL/EMPTY/FORMAT) in allen Kontexten.

## Further Considerations

1. **simpleString in verschiedenen Kontexten**: `simpleString` kann `nameOrKeyword` (NAME/BOOL/NULL/EMPTY/FORMAT) oder `quotedString` sein. In `templateParameterDelegate` wird `nameOrKeyword` als `Name.Variable` tokenisiert (per [tokens.md](tokens.md) Zeile 24), während in `configDefinitionString` der Wert `simpleString` → `String` ist (per [tokens.md](tokens.md) Zeile 12). Da der Lexer ohne vollen Parser-Kontext nicht unterscheiden kann, ob ein NAME Teil von `templateParameterDelegate` oder `configDefinition` ist, approximiert er: Namen nach `=` oder nach COLON werden kontextabhängig tokenisiert.

2. **Disambiguation durch State-Tracking**: Der Lexer nutzt separate States (`parameter-format-colon`, `parameter-format-name`) um die 3-Token-Sequenz von `parameterFormat` korrekt zu handhaben, und erkennt in `template-body` das Pattern `Name.Variable` `=` `Name.Variable` für `templateParameterDelegate` vs. `Name.Attribute` `:` für `configDefinition`.
